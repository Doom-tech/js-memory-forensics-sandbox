# JS Memory Forensics Sandbox

Это мой учебный проект на C++ про безопасный разбор подозрительного JavaScript.
Идея простая: иногда в фишинговом письме нужная злоумышленнику строка не лежит в
файле открытым текстом. Скрипт собирает её только во время выполнения: через
`String.fromCharCode`, xor, `atob`, конкатенацию, `eval` и похожие приёмы.

Обычный поиск по исходнику в такой ситуации может ничего не найти. Поэтому я
сделал маленькую песочницу: она запускает JS в отдельном V8-контексте, а потом
пытается найти строки, которые появились уже после выполнения кода.

## Что умеет программа

Проект делает такой пайплайн:

1. запускает JavaScript в пустом V8-контексте без Node.js, сети и файловой системы;
2. ограничивает выполнение по времени и по размеру V8 heap;
3. на Linux читает карту памяти процесса из `/proc/self/maps`;
4. копирует читаемые регионы памяти через `process_vm_readv`;
5. достаёт из дампов ASCII и UTF-16LE строки;
6. отдельно пишет подозрительные строки с `eval(`, `fetch(`, `http://`,
   `powershell`, `ActiveXObject` и другими индикаторами;
7. сохраняет V8 heap snapshot, чтобы можно было посмотреть граф JS-объектов.

Ещё есть функция `mark(name, value)`. Её можно вызвать прямо из анализируемого
скрипта, чтобы пометить интересную строку или объект. Программа закрепит это
значение в памяти и потом попробует найти его байты в raw dump.

## Важное ограничение

V8 через публичный API не даёт удобную таблицу вида `JS object -> адрес объекта в
памяти`. Поэтому здесь сделан более практичный вариант для учебного проекта:

- `heap_snapshot.json` показывает объекты, строки и связи на уровне V8;
- `mark(name, value)` помогает явно отметить то, что интересно аналитику;
- `tracked_locations.tsv` показывает, где в дампах нашлись байты отмеченного значения.

Для совсем точного адресного маппинга всех объектов пришлось бы собирать debug
V8, патчить аллокатор или лезть во внутренние API V8. Для этой работы я решил
оставить реализацию понятной и объяснимой.

## Зависимости

Для сборки нужны:

- CMake 3.20+;
- компилятор C++20;
- Doxygen, потому что HTML-документация собирается автоматически;
- V8 development package для реального запуска JS.

На macOS я ставил зависимости так:

```bash
brew install v8 doxygen
```

Если V8 не найден, проект может собраться с диагностической заглушкой. Но для
проверки полноценной версии лучше включить строгий режим:

```bash
cmake -S . -B build -DJSFS_REQUIRE_V8=ON
```

## Сборка

```bash
cmake -S . -B build -DJSFS_REQUIRE_V8=ON
cmake --build build
```

Во время обычной сборки CMake также запускает Doxygen. Готовая HTML-документация
появляется в:

```text
docs/api/html/index.html
```

## Тесты

```bash
ctest --test-dir build --output-on-failure
```

Сейчас зарегистрировано 10 отдельных тестов. Они проверяют:

- base64-декодер для `atob`;
- обработку пробелов и padding в base64;
- ошибки на битом base64;
- парсинг нормальной строки maps;
- отбрасывание битых строк maps;
- чтение maps-файла;
- фильтр подозрительных строк;
- извлечение ASCII и UTF-16LE строк;
- поиск значений из `mark(name, value)`;
- отрицательные сценарии с отсутствующими файлами и неправильными параметрами.

## Как запустить пример

Без дампа памяти, но с реальным V8 и heap snapshot:

```bash
./build/js-memory-sandbox \
  --out=analysis_out \
  --no-memory-dump \
  samples/harmless_obfuscated.js
```

Полный вариант на Linux:

```bash
./build/js-memory-sandbox \
  --out=analysis_out \
  --timeout-ms=2000 \
  --max-old-space-mb=64 \
  --max-region-mb=128 \
  --skip-file-backed \
  samples/harmless_obfuscated.js
```

На macOS часть с `/proc/self/maps` не работает, потому что это Linux-механизм.
Зато V8-запуск, тесты и heap snapshot работают нормально.

## Что получается на выходе

После анализа создаётся папка `analysis_out`:

```text
analysis_out/
├── report.md
├── strings.tsv
├── hits.tsv
├── tracked_locations.tsv
├── heap_snapshot.json
└── memory/
    ├── region_0000_...
    └── region_0001_...
```

Самые полезные файлы:

- `report.md` — общий итог запуска;
- `strings.tsv` — все найденные ASCII/UTF-16LE строки;
- `hits.tsv` — только подозрительные строки;
- `tracked_locations.tsv` — где нашлись значения, отмеченные через `mark`;
- `heap_snapshot.json` — снимок heap в формате V8.

## Мини-API внутри JS

В глобальном объекте специально оставлены только небольшие helper-функции:

```js
print("message");
mark("payload", payload);
const decoded = atob("SGVsbG8=");
```

Там нет `require`, `process`, `fetch`, `XMLHttpRequest`, доступа к файловой
системе и сетевых API. Это не делает запуск настоящей малвари полностью
безопасным, но сильно сужает поверхность внутри JS-контекста.

## Безопасность

Это учебный defensive-инструмент, а не промышленная malware-lab. Реальные
подозрительные файлы всё равно лучше запускать только в отдельной VM или
контейнере без личных данных и без доступа к рабочей сети.
